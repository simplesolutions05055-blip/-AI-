# תוכנית E2E לעמודי הפקת תוצרים

## המטרה

להחליף את חוויית “תכתוב בצ׳אט ונבין לבד” בחוויה מסודרת יותר: לכל סוג תוצר יהיה עמוד ייעודי עם טופס, בריף לאישור, עד 3 סבבי תיקון, ואז הפקה ושליחה במייל לפי בחירה.

הצ׳אט הקיים לא נמחק ולא מוחלף. הוא נשאר לשימוש עתידי, לסימולציות ולתהליכים פתוחים יותר.

## סוגי עמודים

לפי מה שקיים במערכת היום, נבנה 4 עמודי הפקה:

1. **הפקת תמונה**  
   לשימושים כמו פוסט, מודעה, הזמנה, גרפיקה לרשתות, תמונת קמפיין.

2. **הפקת מצגת**  
   למצגת עסקית, שיווקית, עירונית או פנימית. המערכת כבר יודעת להפיק מבנה מצגת ותוכן שקפים.

3. **הפקת טקסט**  
   למשל פוסט, הודעה, מייל, נאום, תוכן שיווקי או בריף כתוב.

4. **הפקת PDF / מסמך**  
   כשרוצים תוצר סופי כקובץ מסודר ולא רק טקסט במסך.

באתר זה יופיע תחת אזור חדש בשם **הפקת תוצרים**.

## איך זה ייראה באתר

במקום להתחיל מצ׳אט, המשתמש יראה מסך בחירה:

**מה תרצה להפיק?**

במסך יהיו כרטיסים ברורים:

- תמונה
- מצגת
- טקסט
- PDF / מסמך

כל כרטיס מוביל לעמוד ייעודי.

לדוגמה, בעמוד “הפקת תמונה” יהיו שדות כמו:

- מה מטרת התמונה?
- איפה היא תפורסם? פייסבוק / אינסטגרם / וואטסאפ / אתר / אחר
- קהל יעד
- טקסט שחייב להופיע בתמונה
- סגנון רצוי
- צבעים / מיתוג
- מידות או יחס תמונה
- דברים שאסור שיופיעו
- האם לשלוח במייל בסוף?
- כתובת מייל, רק אם בחרו לשלוח

בעמוד “הפקת מצגת” יהיו שדות כמו:

- נושא המצגת
- קהל יעד
- מטרת המצגת
- מספר שקפים רצוי
- טון: עסקי / שיווקי / רשמי / פשוט
- נקודות שחייבות להופיע
- חומרי מקור אם יש
- האם לשלוח במייל בסוף?

## הזרימה העסקית

כל עמוד יעבוד באותה שיטה:

1. **מילוי טופס**  
   המשתמש ממלא את הפרטים הרלוונטיים לתוצר.

2. **יצירת בריף**  
   המערכת לא מפיקה מיד את התמונה או המצגת. קודם היא הופכת את הטופס לבריף מקצועי.

3. **הצגת בריף לאישור**  
   המשתמש רואה סיכום ברור:
   - מה הולכים להפיק
   - למי זה מיועד
   - מה חייב להופיע
   - מה הסגנון
   - מה המידות או המבנה
   - מה יישלח במייל, אם בכלל

4. **אישור או תיקון**  
   המשתמש בוחר:
   - “מאשר, תפיק”
   - “צריך תיקון”

5. **עד 3 תיקוני בריף**  
   אם המשתמש מתקן, הוא כותב מה לשנות.  
   המערכת מייצרת בריף מעודכן.  
   אחרי 3 סבבים, היא מציגה הודעה עסקית פשוטה:  
   “הגענו למקסימום תיקוני בריף. אפשר להפיק עכשיו או להתחיל מחדש.”

6. **הפקה**  
   אחרי אישור הבריף, המערכת מפעילה את מנוע ההפקה המתאים:
   - תמונה → `generate-image`
   - מצגת → `generate-presentation`
   - טקסט/PDF → הצינור הקיים לפי סוג התוצר

7. **תצוגת תוצאה**  
   בסיום המשתמש רואה:
   - תצוגה מקדימה של התוצר
   - כפתור הורדה
   - סטטוס QA / בדיקה אם קיים
   - אפשרות לשלוח במייל אם לא נבחר קודם

8. **שליחה במייל**  
   אם המשתמש בחר לשלוח:
   - המערכת מבקשת או מאשרת כתובת מייל
   - מפעילה את `send-output`
   - מציגה סטטוס: נשלח / נכשל / דורש טיפול

## מה צריך לבנות בפועל

1. **ניווט חדש**  
   להוסיף בתפריט האדמין אזור בשם “הפקת תוצרים”.

2. **עמוד בחירת סוג תוצר**  
   מסך ראשי עם כרטיסים: תמונה, מצגת, טקסט, PDF.

3. **עמודי טופס ייעודיים**  
   עמוד נפרד לכל תוצר, עם שדות שמתאימים אליו.

4. **רכיב משותף לזרימת אישור**  
   אותו רכיב ישרת את כל סוגי התוצרים:
   - טופס
   - בריף
   - תיקונים
   - אישור
   - הפקה
   - תוצאה
   - שליחה במייל

5. **שמירת מצב**  
   כל בקשת הפקה תישמר כ־request במערכת, כדי שיהיה אפשר לראות אותה גם במסכי הבקשות והתוצרים הקיימים.

6. **התאמה לצינור הקיים**  
   לא להחליף את מנועי ההפקה. להשתמש במה שכבר קיים:
   - `process-request`
   - `generate-image`
   - `generate-presentation`
   - `send-output`

7. **מגבלת 3 תיקונים**  
   לשמור מונה תיקוני בריף לכל בקשה.  
   אחרי 3 תיקונים לחסום תיקון נוסף ולהציע הפקה או התחלה מחדש.

## התוצאה למשתמש

במקום שיחה פתוחה ולא תמיד צפויה, המשתמש מקבל תהליך ברור:

ממלאים פרטים → מקבלים בריף → מאשרים או מתקנים → מפיקים → מורידים או שולחים במייל.

זה ירגיש יותר כמו מערכת הפקה מקצועית ופחות כמו צ׳אט. הצ׳אט נשאר במערכת, אבל ברירת המחדל באתר תהיה עמודים מסודרים לפי סוג תוצר.
