# תיעוד סשן — Flow חדש לבוט הוואטסאפ (תפריטים, זיהוי, תיקונים, תזמון)

**תאריך:** 03/07/2026
**היקף:** תכנון ויישום E2E של flow שיחה חדש ב-WhatsApp/Twilio, כולל זיהוי משתמש לפי טלפון, תפריט תוצרים, תפריט פעולות אחרי מסירה, פענוח כוונות עם AI, תזמון פרסום לרשתות, כיסוי מקרי קיצון וצמצום כמות ההודעות.

---

## 1. מה היה קודם

- כל הודעה נכנסה לזרימה חופשית: AI ניתח את הטקסט וניחש לבד את סוג התוצר.
- אין זיהוי משתמש — שיחות WhatsApp לא היו מקושרות לחשבון באתר, בקשות נוצרו בלי `created_by` ובלי מותג.
- אחרי מסירה השיחה נסגרה; אין תיקון מובנה ואין תזמון.
- מסירת תמונה = 5 הודעות (אישור קבלה, "אני עובד", תמונה עם פוטר צבעים ארוך, טקסט פוסט, הודעת סיום).

## 2. ה-Flow החדש

### זיהוי לפי טלפון
- המספר מנורמל (`+972…` ↔ `05…`, פונקציית `normalizePhone`) ומושווה ל-`profiles.phone`.
- נמצא → השיחה מקושרת (`conversations.user_id`), כל בקשה מקבלת `created_by` + `brand_id` מ-`user_brands` (מותג יחיד למשתמש) → משתקף באתר (קבצים, בקשות, עלויות, גאנט) ומדלג על שאלת "איזו רשות?".
- לא נמצא → הודעת הפניה להרשמה (קישור מ-setting בשם `site_url`, ברירת מחדל `https://ai-mggo.vercel.app`), עם **throttle של 6 שעות** כדי לא לחפור.
- באונבורדינג באתר: שדה טלפון הפך **חובה**, עם ולידציה למספר ישראלי, נשמר מנורמל, וכיתוב שמסביר שהוא משמש לזיהוי בבוט.

### תפריט ראשי
```
היי איתי, מה תרצה להכין היום? 👋
1️⃣ תמונה / פוסט
2️⃣ מצגת
3️⃣ מסמך
```
- הבחירה **נועלת** את סוג התוצר (`output_type_locked`) — ה-worker דורס את הניחוש של המנתח.
- טקסט ארוך במקום בחירה → נופל לזרימה החופשית הישנה (לא נתקעים).

### מסירה (כלכלת הודעות)
ליצירת תמונה+פוסט — **3 הודעות בלבד** במקום 5:
1. `קיבלתי! מכין את התמונה והפוסט — זה לוקח בערך דקה ⏳` (ack יחיד; דגל `ack_sent` בבריף גורם ל-worker לדלג על ה-preflight שלו)
2. **התמונה עם טקסט הפוסט כ-caption** (הודעה אחת = תמונה + מלל; פוסט מעל ~950 תווים נשלח כהודעה נפרדת כי WhatsApp מגביל caption ל-1024)
3. תפריט הפעולות
- פוטר הצבעים/מיתוג **הוסר** לגמרי.
- אחרי שליחת מדיה יש השהיה של 6 שניות לפני הטקסטים הבאים — Twilio מוסר מדיה לאט מטקסט, בלי זה התפריט הגיע לפני התמונה.

### תפריט פעולות אחרי מסירה (שטוח, הודעה אחת)
לתמונה:
```
1️⃣ לתקן את התמונה עם AI ✏️
2️⃣ לתקן את טקסט הפוסט 📝
3️⃣ לתזמן פרסום 📅
4️⃣ תוצר חדש ✨
אפשר להשיב במספר, או פשוט לכתוב מה לשנות — ואני כבר אבין 🙂
```
לתוצר שאינו תמונה: 3 אפשרויות (תיקון / תזמון / חדש).

### פענוח טקסט חופשי עם AI
- פונקציה חדשה `classifyPostDeliveryIntent` ב-`openai.ts`: מסווגת הודעה חופשית אחרי מסירה ל-`image_fix / caption_fix / content_fix / schedule / new_request / unclear`.
- "תחליף את הרקע לכחול" → עריכת התמונה רצה **מיד**, בלי שאלה נוספת.
- "תקצר את הפוסט" → ניסוח מחדש של המלל מיד.
- בריף חדש → עובר לפייפליין יצירה מלא. לא ברור → תצוגת התפריט שוב.
- ברכות/תודה ("תודה", "מעולה", 👍) מזוהות דטרמיניסטית — בלי קריאת LLM מיותרת.

### תיקונים
- **תמונה:** קריאה לפונקציית `edit-image` הקיימת (img2img) עם המשוב; רץ ברקע (~דקה), התוצאה נשלחת כתמונה חדשה ומשתקפת באתר. `edit-image` מקבלת עכשיו `created_by` לייחוס.
- **מלל פוסט:** `generateSocialCaption` במצב revision; מעדכן את `outputs.text_content` (מה שגם מוזן לתזמון).
- **מסמך/מצגת/טקסט:** בקשת revision חדשה (סטטוס `queued`, בריף מוכן + המשוב ב-`admin_note`) שרצה בפייפליין הרגיל.

### תזמון פרסום
- פלטפורמה (פייסבוק / אינסטגרם / שניהם) → תאריך ושעה בעברית ("מחר 10:00", "25/12 18:00", שעון ישראל עם המרת DST נכונה ל-UTC) → אישור.
- נכנס ישירות ל-`scheduled_social_posts` (שורה לכל פלטפורמה, עם המדיה, הכיתוב, המותג והמשתמש) → מופיע מיד בגאנט התזמונים באתר.
- אינסטגרם בלי תמונה נחסם (דורש מדיה). הפרסום בפועל למטא — פרויקט נפרד עתידי.

## 3. מקרי קיצון שכוסו

| מקרה | התנהגות |
|---|---|
| מספר לא מוכר | הודעת הרשמה, throttle 6 שעות (לא בכל הודעה) |
| לחיצה כפולה על אפשרות בתפריט ("1" ואז שוב "1") | הצגה חוזרת של בקשת הבריף — לא נוצר תוצר מהטקסט "1" |
| בריף קצר מדי ("אוקי", אימוג'י) | בקשת פירוט במקום שריפת יצירה על זבל |
| "בטל" / "תפריט" בזמן שממתינים למשוב תיקון | ביטול וחזרה לתפריט — לא נשלח כמשוב לעריכת AI |
| ברכה/תודה אחרי מסירה | תפריט הפעולות שוב, בלי קריאת LLM |
| הודעות כפולות מ-Twilio (retries) | dedup לפי `MessageSid` בכל שלב |
| הודעות בזמן יצירה (burst) | debounce קיים + הודעת "כבר בטיפול" **פעם אחת ב-3 דקות** במקום על כל הודעה |
| כשל יצירה (OpenAI down / quota) | המשתמש מקבל הודעת שגיאה ברורה, והשיחה **משוחררת** — ההודעה הבאה פותחת בקשה חדשה (קודם: שקט נצחי) |
| needs_attention (תקרת תקציב / סבבי תיקון / QA) | ההודעה למשתמש נשלחת + השיחה משוחררת — לא נתקעים מול בקשה "סגורה" |
| פוסט ארוך מ-950 תווים | נשלח כהודעה נפרדת (מגבלת caption של WhatsApp) |
| שינוי דעת בזמן בריף ("מצגת" אחרי שבחר תמונה) | מחליף סוג ומציג את הפרומפט המתאים |
| מייל אחרי מסירה | ממשיך לעבוד — שליחת עותק למייל (flow קיים) |
| reset ("התחל מחדש") בכל שלב | איפוס מלא + תפריט ראשי |
| תאריך תזמון בעבר / לא תקין | הודעת שגיאה ידידותית ובקשה חוזרת |
| soft-close אחרי חוסר פעילות (סשן מקביל) | השיחה נשמרת כ-`soft_closed`; חזרה מציעה "להמשיך / תוצר חדש" |

## 4. קבצים שהשתנו

| קובץ | מה |

| `supabase/functions/_shared/flow.ts` | **חדש** — כל מכונת המצבים: זיהוי, תפריטים, תיקונים, תזמון, פרסינג, שעון ישראל |

| `src/pages/OnboardingPage.tsx` | טלפון חובה + ולידציה + נרמול + כיתוב הסבר |
| `supabase/migrations/20260703090000_whatsapp_flow_menu.sql` | עמודות flow בשיחות: `user_id`, `flow_state`, `flow_context`, `selected_output_type`, `last_delivered_request_id` |
| `supabase/migrations/20260703164728_whatsapp_soft_close_artifact_lifecycle.sql` | (סשן מקביל) סטטוס `soft_closed` ומחזור חיים של תוצר |

## 5. מצבי ה-Flow (`conversations.flow_state`)

`main_menu` → `awaiting_brief` → (יצירה) → `post_delivery` → `awaiting_image_fix` / `awaiting_caption_fix` / `awaiting_fix_feedback` / `schedule_platform` → `schedule_datetime` → `schedule_confirm` → חזרה ל-`post_delivery`. `NULL` = זרימה חופשית ישנה.

## 6. פריסה ובדיקות

- **Migrations הוחלו** (`supabase db push --linked`), **6 פונקציות נפרסו**: twilio-webhook, simulator-message, process-request, send-output, conversation-maintenance, edit-image.
- נבדק E2E בסימולטור (אותו מנוע של WhatsApp): flow מלא לתמונה (3 הודעות), פענוח AI לתיקון מלל ולתיקון תמונה, תזמון לשתי רשתות עם אימות שורות ב-DB (שעה ישראלית→UTC נכון), revision למסמך, reset, לחיצה כפולה, בריף קצר, "בטל" בתיקון, "תודה" אחרי מסירה.

## 7. ידוע / להמשך

- **PDF שבור מלפני הסשן:** `renderPdfBase64` נכשל על `Missing APP_URL / INTERNAL_API_SECRET` — נפתחה משימה נפרדת להגדרת הסודות.
- כפתורי WhatsApp אמיתיים (Quick Reply) דורשים Content Templates ב-Twilio — כרגע תפריטים ממוספרים.
- פרסום בפועל לפייסבוק/אינסטגרם (טוקני Meta + worker) — פרויקט נפרד; התזמון נרשם בגאנט כ"מתוזמן".
- מצגת נמסרת כראשי פרקים בטקסט (לא PPTX) — לפי החלטה מפורשת.
- כדי שהבוט יזהה משתמש קיים, הטלפון שלו חייב להיות מעודכן בפרופיל באתר.
